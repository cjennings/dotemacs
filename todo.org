It turns out that nothing mystical is happening in your =dupre-theme.el= that would prevent the Emacs server from creating its socket.  When =emacsclient= says:

    emacsclient: can't find socket; have you started the server?

it really does mean ‚Äúthere is no running Emacs ‚Äìdaemon laying down =<‚Ä¶>/server/= for me to talk to.‚Äù  By itself, loading a custom theme (even one with a few Unicode ellipses or =display-graphic-p= branches) will not stop the server socket from being created.  What is much more often the culprit is:

  1.  Mis-using =emacsclient -a ''= so that it never actually starts the server, or  
  2.  Having a startup error in your init file (or theme) that aborts =server-start= before the socket ever gets written.

Here is how I‚Äôd go about diagnosing (and fixing) the situation:

1)  Make sure you can start Emacs ‚Äìdaemon cleanly, /without/ any of your custom code.

   From a shell, run

      emacs --daemon --debug-init --eval "(message \"daemon ok\")"

   ‚Äì you should see a process stick around, and if you look at =~/.emacs.d/server/= you should find a valid socket file.

   If that does not happen, Emacs itself is aborting in your init.  Remove or comment out your calls to =load-theme=, your theme-specific code, all your =custom-set-‚Ä¶= blocks, and any other ELisp until you can get a clean daemon start.  Once that is rock-solid, re-introduce your theme.

2)  Once you can start the daemon by hand, verify that =emacsclient -c= can actually talk to it:

      emacsclient -c -n

   The =-n= is ‚Äúdon‚Äôt wait for the client buffer to finish,‚Äù so you‚Äôll get your GUI frame immediately.  If that works, the server is up and your sockets are good.

3)  If you /really/ want =emacsclient -c -a ''= to both start Emacs ‚Äìdaemon /and/ immediately pop a graphical frame, you have two choices:

   ‚Ä¢  Let =-a= fall back to =emacs --daemon=.  On many platforms simply omitting the =-a= will do this.  
   ‚Ä¢  Explicitly tell it:

         emacsclient -c -a "emacs --daemon"  # note the command in quotes

   On macOS/Homebrew, for instance, people often do

         alias ec='emacsclient -c -a "emacs --daemon"'

   so that if there is no server, emacsclient will shell out to =emacs --daemon=, launch it, and /then/ connect.  

4)  If you still suspect your theme is triggering a startup error (and killing the daemon before it gets to =(server-start)=), you can sprinkle a tiny bit of ELisp into your init:

   (setq debug-on-error t)
   (add-hook 'after-init-hook
             (lambda ()
               (message "finished init; about to (server-start)")))

   (unless (server-running-p)

      emacs --daemon --debug-init

      emacs -Q

      M-x load-file RET /path/to/dupre-theme.el RET
